## Phase 5 — Backend: Quiz-Taking Engine & Architecture Restructure

Everything on the Laravel side, done in one pass since no real data existed yet — full freedom to restructure rather than layer patches on top of the Phase 2 (local) work above. See [`apps/api/docs/ARCHITECTURE.md`](../apps/api/docs/ARCHITECTURE.md) for the resulting structure as a living reference; this section is the log of how it got there.

- [x] **API versioning + restructure** — all routes moved under `/api/v1`; controllers/requests moved from `Api\Admin\*` / `Api\*` into `Api\V1\{Admin,Auth,Public}\*` namespaces (paths under `/admin/*` unchanged otherwise). `apps/web/lib/api.ts`, `.env.local`, and `.env.example` updated to the new base URL.
- [x] **Layering** — business logic pulled out of the admin controllers into single-purpose `app/Actions/Quiz/*` classes (`GenerateUniqueSlug`, `SyncQuizQuestionAnswers`, `CreateQuizQuestion`, `UpdateQuizQuestion`, `GradeQuizAttempt`). Every public-facing response now goes through an API Resource (`app/Http/Resources/Api/V1/...`) instead of a raw model — verified via curl that pre-submission questions never leak `is_correct`/`explanation`, and that grading reveals them correctly after submit. Added `QuizPolicy` (`view()` checks `is_active`; Phase 5 of the product roadmap will extend it for `is_premium`).
- [x] **Media**: replaced hand-rolled `cover_image_path`/`images` columns + manual `Storage::delete()` bookkeeping with **Spatie Media Library** (`spatie/laravel-medialibrary`). `Quiz` gets a single-file `cover` collection, `QuizQuestion` an `images` collection; deleting either model now auto-deletes its files. `cover_image_url`/`image_urls` stay as computed attributes so the JSON shape didn't change for existing consumers.
  - **Contract change to flag**: admin question-image editing's `keep_images` field (array of file paths) is now `keep_media_ids` (array of numeric media IDs), since media library assigns IDs rather than stable paths. `apps/web`'s admin quiz-question edit page still expects the old shape and needs a small update — not yet done, out of scope for this backend pass.
- [x] **Migration squash + policy**: combined the incremental `add_images_to_quiz_questions_table` / `add_order_no_to_quizzes_table` / `add_cover_image_path_to_quizzes_table` / `add_explanation_to_quiz_answers_table` / `add_state_vehicle_track_to_quizzes_table` / `add_is_admin_to_users_table` migrations into their base `create_*` migrations for a clean history — safe only because nothing had run against real data yet. Documented in `apps/api/docs/ARCHITECTURE.md` as a **one-time move, not a standing practice**: every new column from here on (including package-added ones — Cashier in a later phase will add its own columns to `users` via its own migration) gets a new, additive migration.
- [x] **Schema hardening**, found during a follow-up review:
  - `quizzes.state_id`/`vehicle_type_id` were nullable in the DB but already `required` in the Store/Update Form Requests — schema and validation disagreed. Made both `NOT NULL` with `restrictOnDelete()`.
  - `test_track` was a bare string column validated with `in:permit_test,driving_test`, inconsistent with how `difficulty`/`status` are proper backed enums elsewhere. Added `App\Enums\TestTrack`, cast it on `Quiz`, switched both Requests to `Rule::enum(TestTrack::class)`.
  - `quizzes.quiz_category_id` used `cascadeOnDelete()` even though `QuizCategoryController::destroy()` already blocks deleting a category with quizzes attached — the FK didn't match that rule, so any path bypassing the controller (tinker, a console command) could've silently wiped every quiz in a category. Changed to `restrictOnDelete()` so the DB enforces it too.
  - `quiz_attempts.quiz_id` was `cascadeOnDelete()` — deleting a quiz would've silently erased everyone's attempt/progress history for it, undermining the roadmap's Phase 4 (progress tracking). Changed to `restrictOnDelete()`.
  - Fixing the NOT NULL constraint surfaced a real bug: `QuizFactory` was setting `state_id`/`vehicle_type_id` to `null` (worked under the old nullable schema, broke under the new one). Fixed it; added `HasFactory` + factories to `State` and `VehicleType`, which didn't have them yet.

## Admin attempt visibility & public header auth-state fix

Two related pieces of follow-up work, done together: giving admins visibility into *other* users' quiz results (until now, `GET /attempts` only ever returned the caller's own history — an admin had no way to inspect a specific learner's performance beyond the aggregate `Admin\StatsController` totals), and a bug found while building the admin page's "Browse Quizzes" sidebar entry.

- [x] **`Api\V1\Admin\AttemptController::index`** — new admin-only endpoint, every user's attempts, filterable by `user_id`/`quiz_id`/`status`. Extracted the shared `quiz.category`/`quiz.quizType`/`answers.question.answers` eager-load into `QuizAttempt::scopeWithReviewDetails()` so it isn't duplicated between this and the existing self-service `Api\V1\QuizAttemptController::index` — the only difference between the two queries is the base (`QuizAttempt::query()` vs. `$request->user()->quizAttempts()`). `QuizAttemptResource` gained a `user` field (nested `id`/`name`/`email`, `null` for guest attempts) reused by both endpoints.
  - Verified query efficiency with a seeded throwaway test (15 attempts × 60 questions each, 900 answer rows): 9 SQL queries total regardless of question count — confirms the eager-load prevents the N+1 that `QuizAttemptAnswerResource` would otherwise cause (it reads `$this->question->question_text` and `$this->question->answers->firstWhere(...)` per answer). Same test measured the response payload at ~190KB for that one page — the per-question answer breakdown isn't paginated separately or capped, so a page of large-question-count attempts is proportionally heavier; flagged as a possible future optimization (defer `answers` to the existing per-quiz "Review" fetch instead of shipping it inline) rather than fixed now, since the self-service endpoint already makes the same tradeoff.
- [x] **Admin UI**: `/admin/attempts` page (`AdminGuard`-wrapped), filters by user and quiz **name** via plain `<select>` dropdowns (populated from the existing `/admin/users`/`/admin/quizzes` list endpoints, no new backend filter needed) rather than raw IDs — chosen over a search-as-you-type combobox since no such component exists yet in this codebase and the plain-dropdown approach needed no new UI primitive or backend search filter. Extracted the attempt-row-with-expandable-review markup out of `dashboard/attempts/page.tsx` into a shared `components/quiz/AttemptRow.tsx` (`showUser` prop toggles the extra name/email line) so the self-service and admin pages don't duplicate it. Sidebar gained an admin-only "All Results" entry.
