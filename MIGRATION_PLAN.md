## shadcn/ui retrofit (correction, applied across Home/Auth/Dashboard)
Initial ports of Accordion, Avatar, Checkbox, Input, Label, Dropdown (UserMenu), and the Dashboard Sidebar were hand-rolled as simplified custom components instead of using **shadcn/ui** — the direct React equivalent of the shadcn-vue primitives the old Vue project actually used (both built on the same underlying pattern: Vue's on Radix-Vue/reka-ui, React's on Radix UI / base-ui). That was a judgment mistake, not a deliberate choice — corrected after the user caught it.
- Ran `npx shadcn@latest init` + `npx shadcn@latest add accordion avatar checkbox input label dropdown-menu breadcrumb sidebar` in `apps/web`. Added `clsx`/`tailwind-merge`/`class-variance-authority`/`tw-animate-css`, `lib/utils.ts` (`cn()` helper).
- **Real complication hit and fixed**: this shadcn CLI version defaults to a `base-ui`-based style, and on Windows' case-insensitive filesystem its lowercase `button.tsx`/`checkbox.tsx`/etc. writes collided with this project's PascalCase `Button.tsx`/`Checkbox.tsx` — silently clobbering the custom `Button` (used everywhere with its own `variant`/`icon`/`href`/`text` API) multiple times via dependency re-pulls. Fixed by restoring `Button.tsx` from the Vue source and giving shadcn's own button a non-colliding name (`components/ui/ShadcnButton.tsx`), repointed only where shadcn's other primitives (`sidebar.tsx`, `sheet.tsx`) require it internally.
- `components/ui/{Button,Heading,Paragraph,PlaceholderPattern,InputError,SocialIcons}.tsx` were **not** touched — they're the project's own hand-built design-system components in both the Vue and React versions, never shadcn ones.
- `tw-animate-css` is now actually imported (`@import "tw-animate-css"` in `globals.css`) — the old Vue project had this package installed but its import commented out, so the Accordion's open/close animation never worked there; it works correctly here. A genuine improvement, not a fidelity break.
- `lib/sidebar-context.tsx` deleted — shadcn's `SidebarProvider` manages its own (cookie-based) open/collapsed state, closer to how Vue's version worked (server-injected cookie) than the localStorage approach it replaced.

## Post-migration bug fixes (found via user testing)
- **Header vehicle selector didn't navigate**: `components/Header.tsx`'s `selectVehicle()` only updated `selectedVehicle` state — Vue's `WebHeader.vue` also calls `router.visit(...)` to actually navigate to the newly-selected vehicle's path (`/{state}`, `/{state}/cdl`, `/{state}/motorcycle`) when a state is already selected. Missed when the Header was first ported during the Home page phase. Fixed: added `useRouter()` + `router.push()` matching Vue's exact guard (`if (!selectedState) return;`) and path logic. Verified: tsc/eslint clean, dev server still 200 on `/oregon`.
- **`written-test/Index.vue`'s "sideDescr" block was dropped entirely, not just simplified**: the fork building `state`/`written-test`/`motorcycle` said it "simplified" some decorative chrome, but a whole functional chunk between the hero thumbnail and the "Tricky exam topics" heading was actually missing outright — the overlapping driver's license sample image, the "Moderate / 6 min / Avg. first-try score: 65%" stat chips (with hover tooltips), and the 3 FMCSA/Police/GHSA partner logos. Caught by the user comparing a screenshot against the live page, not by any automated check (tsc/eslint/build all stayed green throughout since this was a missing DOM subtree, not a type or syntax error — a reminder that green CI here only proves the code compiles, not that content fidelity is complete). Rebuilt the full block in `components/state/WrittenTestContent.tsx` matching Vue's markup/order (including the CSS `order-[-1]` trick that visually reorders the stat chips above the thumbnail despite DOM order), reusing the `partner_gray_{1,2,3}.svg` assets the fork had already copied but never referenced. The license image intentionally hotlinks to the same external `driving-tests.org` CDN URL Vue does (hardcoded to an Alabama sample image regardless of which state's page it is — that's Vue's actual behavior, not a bug, kept as-is for fidelity). Verified: tsc/eslint clean, fresh `pnpm build` succeeds (27 routes, unchanged), live content check confirms "Moderate", "6 min", "Avg. first-try score: 65%", and the license image URL all present in the rendered HTML, and all 3 partner logo assets return 200.
